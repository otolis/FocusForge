**1. Scaffold Flutter web platform support:**

The project currently has NO `web/` directory. Flutter web builds require a `web/index.html` and supporting files. Run:

```bash
flutter create --platforms web .
```

This generates the `web/` directory with:

- `web/manifest.json` (PWA manifest)

- `web/icons/` directory (PWA icons)

After scaffolding, verify the files exist:

**IMPORTANT:** The `flutter create --platforms web .` command will NOT overwrite any existing files in `lib/`, `pubspec.yaml`, etc. It only generates the missing `web/` directory.

**2. Modify lib/features/smart_input/presentation/providers/smart_input_provider.dart:**

Add the kIsWeb import at the top of the file:
```dart
import 'package:flutter/foundation.dart' show kIsWeb;
```

Modify the `smartInputInitProvider` to skip TFLite initialization on web. Replace the existing implementation:

FROM:
```dart
final smartInputInitProvider = FutureProvider<void>((ref) async {
  final service = ref.read(smartInputServiceProvider);
  await service.initialize();
});
```

TO:
```dart
/// Initializes the TFLite model. Call once during app startup or feature entry.
///
/// On web, TFLite is not supported (FFI bindings unavailable). The provider
/// returns immediately, and the smart input pipeline falls back to regex-only
/// parsing which works without TFLite.
final smartInputInitProvider = FutureProvider<void>((ref) async {
  if (kIsWeb) return; // TFLite not supported on web; regex parser still works
  final service = ref.read(smartInputServiceProvider);
  await service.initialize();
});
```

The rest of the file (nlpParserProvider, tfliteClassifierProvider, smartInputServiceProvider, smartInputProvider) stays unchanged. The `smartInputProvider` calls `service.parseInput()` which internally checks `_isLoaded` on the TFLite classifier -- if TFLite never loaded (because we skipped init on web), it gracefully falls back to regex-only results. This is the existing behavior documented in the SmartInputService.

No changes needed to any other file. The `tflite_flutter` import in `tflite_classifier_service.dart` is tree-shaken on web -- the import itself doesn't crash, only the native binding calls do. Since `initialize()` is never called on web, those binding calls never execute.

**Create .github/workflows/deploy-web.yml:**

First ensure the .github/workflows/ directory exists (it does not currently exist in the project).

Create the following GitHub Actions workflow file:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Flutter
        uses: subosito/flutter-action@v2
        with:
          channel: stable

      - name: Install dependencies
        run: flutter pub get

      - name: Build Flutter web
        run: flutter build web --release --base-href '/FocusForge/'

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build/web
```

Key details:
- Triggers on push to main branch and manual workflow_dispatch
- Uses `subosito/flutter-action@v2` with stable channel (matches project's Flutter >=3.29.0 requirement)
- `--base-href '/FocusForge/'` is critical for GitHub Pages subpath hosting (trailing slash required)
- `peaceiris/actions-gh-pages@v4` publishes the `build/web/` directory to the `gh-pages` branch
- `permissions: contents: write` is required for the gh-pages action to push to the gh-pages branch
- The repo name in base-href matches the GitHub repository name "FocusForge"
